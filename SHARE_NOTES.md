# 移动开发套件使用分享

## 分享定位

这次分享不是为了推广某个项目，也不是为了安利某个特定工具。

我真正想分享的是：

- 当 Coding Agent 开始改变我们的工作方式之后
- 我们应该开始围绕自己的工作流
- 去做一些真正贴合自己的“妙妙小工具”

这次这个项目，只是一个例子。

---

## 我做这个项目的初心

我发现自己最近在用 `Cursor`、`Codex` 这些 Coding Agent 的时候，工作方式已经发生变化了。

以前开发更多是：

- 自己盯着代码
- 自己一点点写
- 自己频繁切换文件、终端、浏览器

但现在越来越多的时候，我在做的是：

- 给 Agent 一个任务
- 看 Agent 的最终回复
- 判断它做得对不对
- 给它纠偏
- 再让它继续干活

也就是说，我很多时候其实已经不是在“亲自写代码”，而是在“指挥和审阅”。

那我就开始想一件事：

**既然我真正需要的是“收结果 + 发下一条指令”，那我为什么一定要守在电脑前？**

于是我就开始做这个小工具：

**把 Codex、Cursor 这些 Coding Agent 接进飞书，让我在飞书里就能继续推进开发任务。**

---

## 这个项目本质上做了什么

它不是一个新的 IDE，也不是一个巨大的平台。

它做的事情其实非常简单：

1. Agent 跑完任务，把结果回到飞书
2. 我在飞书里继续发指令
3. 它把消息转给对应的 Coding Agent
4. Agent 继续干活

进一步一点，它还支持：

- 多 Session
- Topic / 话题化协作
- 工作区切换
- 工作区 fork
- 附件、图片、文件传递
- 群聊协作
- Session 转移

但这些都只是“工程细节”。

这个项目真正重要的不是功能列表，而是：

**我和 Coding Agent 的协作接口，被重新设计了一遍。**

---

## 这个项目对我意味着什么

### 1. 我不再需要一直守在电脑前

以前移动办公更多是：

- 回消息
- 看文档
- 回应线上问题

但现在，借助飞书里的 Agent 交互，我是真的可以：

- 在路上发指令
- 看结果
- 做审阅
- 给纠偏意见
- 让它继续执行

也就是说，移动办公第一次真正延伸到了“开发协作”。

### 2. 工作界面没有那么重要了，协作接口更重要

以前我们默认 IDE 是开发的主战场。

但现在，当 Agent 已经能承担越来越多的执行工作时，很多时候真正关键的反而是：

- 我怎么发起任务
- 我怎么感知进度
- 我怎么审阅结果
- 我怎么继续纠偏

也就是说，真正重要的开始变成：

**我如何和 Agent 协作。**

### 3. 工具不一定要大，只要贴合自己的工作流就有价值

这个项目并不复杂，也不是一个大而全的平台。

它只是非常准确地解决了我自己的一个问题：

**我想不守在电脑前，也能继续和 Coding Agent 协作。**

很多时候，小工具最有价值的地方，不在于通用，而在于贴合。

以前我们更多是在适应工具。

现在我越来越觉得，我们应该反过来：

**让工具适应我们自己的工作方式。**

我很喜欢把这个比喻成：

## 当罗杰开启大航海时代

大航海时代最有意思的地方，不是海已经在那里，
而是大家开始意识到：

- 原来可以出海
- 原来可以去更远的地方

对我们现在来说也是一样。

Coding Agent 出现之后，我们其实也进入了一个新的时代：

- 不只是“用 AI”
- 而是“围绕 AI 重新设计自己的工作方式”

- 而是“开始做自己的工具”

所以我这次真正想传达的是：

**希望大家都开始构建自己的妙妙小工具，真正地把自己解放出来。**

## 为什么我觉得这件事值得大家都去试

因为现在做这种工具的门槛，真的已经比以前低很多了。

以前你要做一个自己的工具，可能要考虑：

- 前端怎么写
- 后端怎么写
- 接口怎么联调
- 怎么上线

现在很多时候，你只需要：

- 明确自己的真实痛点
- 搭一个很薄的交互层
- 把 Agent 接进去

它可能就已经足够有用了。

也就是说：

**从今天开始，工程师不只是可以写业务，也可以很自然地为自己造工具。**

## 关于 Coding 工具的一点个人观察

我们公司最近也在考虑切换 Coding 工具，之前主要是 `Cursor`。

我自己这段时间实际用下来，一个比较强的感受是：

**`Codex` 真的又快又好。**

尤其是在这种任务驱动的协作方式下，它很适合：

- 直接接任务
- 快速给结果
- 让我审阅和纠偏

所以这个项目后来也不只是接了一个 Agent，而是逐步接入了：

- `Codex`
- `Cursor Agent`
- 以及其他可扩展的 Coding Agent

这样我真正得到的，不是“绑定某个工具”，而是一个统一的协作入口。

---

## 这次项目里最让我有感觉的一点

我后来慢慢意识到：

这个项目的意义，不在于“我把飞书和 Agent 打通了”。

而在于：

**我开始认真承认，自己的工作方式已经变了。**

以前是：

- 我直接操作代码

现在越来越像：

- 我给出目标
- Agent 去执行
- 我来判断方向

这意味着，我的角色正在往“调度者、审阅者、纠偏者”转移。

而这个小工具，正好承接了这种变化。
